Fix GAP-05 (edit lock stuck after remove-stop and other edit operations).

**Root cause:** The `openSSE()` function in api.js calls `handlers.onerror()` when the EventSource connection errors or closes unexpectedly. But none of the edit operation SSE handlers in guide-edit.js register an `onerror` handler. If SSE drops before the completion event, `_unlockEditing()` is never called, permanently blocking all edit buttons.

**Fix:** Add an `onerror` handler to every `openSSE()` call in guide-edit.js that:
1. Closes the SSE connection
2. Calls `_unlockEditing()`
3. Shows an error toast or alert

There are 5 places in guide-edit.js that call `openSSE()`. Add `onerror` to each:

1. **_executeRemoveStop** (~line 109): Add to the handlers object:
```javascript
onerror: () => {
  if (_editSSE) { _editSSE.close(); _editSSE = null; }
  _unlockEditing();
  alert('Verbindung verloren beim Entfernen des Stopps.');
}
```

2. **_executeAddStop** (~line 240-270, the openSSE for add_stop_complete): Same pattern with message `'Verbindung verloren beim Hinzufuegen des Stopps.'`

3. **_doAddStopFromMap** (~line 430-460, openSSE for add_stop_complete from map click): Same pattern with message `'Verbindung verloren beim Hinzufuegen des Stopps.'`

4. **_onStopDrop** (~line 501): Same pattern with message `'Verbindung verloren beim Sortieren.'`

5. **_listenForReplaceComplete** (~line 721, the _replaceStopSSE): Use `_replaceStopSSE` variable instead of `_editSSE`:
```javascript
onerror: () => {
  if (_replaceStopSSE) { _replaceStopSSE.close(); _replaceStopSSE = null; }
  _unlockEditing();
  alert('Verbindung verloren beim Ersetzen des Stopps.');
}
```

All messages are in German per project convention. Each onerror handler follows the exact same pattern as the existing `job_error` handlers in each section.

Fix GAP-04 (click-to-add popup shows garbled characters instead of readable place name).

**Root cause analysis:** The `_onMapClickToAdd` function uses Google Maps Geocoder which returns `formatted_address` — this is typically fine. The garbled characters likely come from one of these issues:
1. The `formatted_address` may contain Unicode characters that display incorrectly
2. The result array may have a more specific result type that gives a cleaner name

**Fix:** Instead of using `results[0].formatted_address` directly, extract a cleaner place name by:
1. First try to find a `locality` or `sublocality` type result (gives city/town names)
2. Fall back to `results[0].address_components` to build a short name (locality + country)
3. Last resort: use `formatted_address` but truncate to first 2 comma-separated parts

Replace the geocoder callback in `_onMapClickToAdd` (lines ~295-299) with:

```javascript
geocoder.geocode({ location: { lat: latLng.lat(), lng: latLng.lng() } }, function(results, status) {
    if (status !== 'OK' || !results || !results.length) return;

    // Try to build a clean short name from address components
    var placeName = '';
    var components = results[0].address_components || [];
    var locality = '';
    var region = '';
    var country = '';
    for (var c = 0; c < components.length; c++) {
      var types = components[c].types || [];
      if (types.indexOf('locality') !== -1) locality = components[c].long_name;
      else if (types.indexOf('administrative_area_level_1') !== -1) region = components[c].long_name;
      else if (types.indexOf('country') !== -1) country = components[c].long_name;
    }

    if (locality) {
      placeName = locality + (country ? ', ' + country : '');
    } else if (region) {
      placeName = region + (country ? ', ' + country : '');
    } else {
      // Fallback: first two parts of formatted_address
      var parts = (results[0].formatted_address || '').split(',');
      placeName = parts.slice(0, 2).join(',').trim();
    }

    _showClickToAddPopup(latLng, placeName);
});
```

This produces clean output like "Annecy, France" instead of a garbled full address.
